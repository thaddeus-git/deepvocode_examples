Git: Could not resolve host github.com error while cloning remote repository in git: Complete Tutorial with 5 Solutions

Quick Answer

The “fatal: unable to access” and “could not resolve host github.com” error typically occurs due to network configuration issues. The fastest fix is to reset your proxy settings:

git config --global --unset http.proxy
git config --global --unset https.proxy

Then restart your terminal and try cloning again.

Choose Your Method

Based on your specific needs:

  • 🚀 Speed Seeker → Method 1: Quick proxy reset (fastest resolution)
  • 📚 Learning Explorer → Method 3: DNS configuration (understand networking fundamentals)
  • 🔧 Problem Solver → Method 5: Complete network reset (guaranteed to work)
  • 🏗️ Architecture Builder → Method 1: Proper proxy configuration (enterprise-ready)
  • 🎨 Output Focused → Method 2: VPN management (clean connection)
  • ⚡ Legacy Maintainer → Method 4: SSL workarounds (compatible with older systems)

Ready-to-Use Code

Quick Fix (Most Common Solution)

# Reset proxy settings that might be interfering
git config --global --unset http.proxy
git config --global --unset https.proxy
 
# Clear any cached credentials
git config --global --unset credential.helper
 
# Restart terminal and try cloning again
git clone https://github.com/username/repository.git

Corporate Network Solution

# Configure proxy for corporate environments
git config --global http.proxy http://proxy.company.com:8080
git config --global https.proxy https://proxy.company.com:8080
 
# Set proxy authentication if required
git config --global http.proxy http://username:password@proxy.company.com:8080

DNS Alternative Solution

# Use SSH instead of HTTPS to bypass DNS issues
git clone git@github.com:username/repository.git

git config --global url."https://140.82.112.3/".insteadOf "https://github.com/"

Method 1: Proxy Configuration Reset (Recommended)

Rating: ⭐ Recommended

How it works:

This method addresses the most common cause of the “could not resolve host” error: misconfigured proxy settings. When Git attempts to connect to GitHub, it may be using outdated or incorrect proxy configurations that prevent proper DNS resolution and network connectivity. By resetting these configurations, we allow Git to use the system’s default network settings.

The error often occurs when:

  • Previous proxy configurations remain active after network changes
  • Corporate VPN software modifies Git settings
  • Manual proxy configuration contains errors
  • Network environment changes but Git retains old settings

Complete Implementation:

#!/bin/bash
# Git Proxy Reset Script - Complete Solution
 
echo "Starting Git network configuration reset..."
 
# Step 1: Display current proxy settings for debugging
echo "Current proxy settings:"
git config --global --get http.proxy
git config --global --get https.proxy
 
# Step 2: Remove all proxy configurations
echo "Removing proxy configurations..."
git config --global --unset http.proxy 2>/dev/null || echo "No http.proxy set"
git config --global --unset https.proxy 2>/dev/null || echo "No https.proxy set"
 
# Step 3: Clear additional network-related configurations
echo "Clearing additional network settings..."
git config --global --unset http.sslVerify 2>/dev/null || echo "No sslVerify override"
git config --global --unset credential.helper 2>/dev/null || echo "No credential helper override"
 
# Step 4: Reset timeout settings to defaults
echo "Resetting timeout configurations..."
git config --global http.lowSpeedLimit 0
git config --global http.lowSpeedTime 999999
 
# Step 5: Test connectivity
echo "Testing GitHub connectivity..."
if curl -s --connect-timeout 10 https://github.com > /dev/null; then
    echo "✅ GitHub is accessible"
else
    echo "❌ GitHub still not accessible - try Method 2"
    exit 1
fi
 
# Step 6: Attempt clone test
echo "Testing git clone functionality..."
git ls-remote https://github.com/octocat/Hello-World.git > /dev/null 2>&1
if [ $? -eq 0 ]; then
    echo "✅ Git clone should work now"
    echo "You can now run: git clone https://github.com/username/repository.git"
else
    echo "❌ Git clone still failing - check network configuration"
fi
 
echo "Proxy reset complete. Please restart your terminal."

Pros:

  • Fixes 80% of “could not resolve host” errors
  • Quick and non-destructive solution
  • Preserves other Git configurations
  • Works across all operating systems
  • No administrative privileges required

Cons:

  • Won’t work if you actually need proxy settings
  • May need to reconfigure legitimate proxy settings
  • Doesn’t address DNS or firewall issues

When to use:

  • First troubleshooting step for connection errors
  • After changing networks (home to office, etc.)
  • When VPN software has been installed/uninstalled
  • After corporate software updates

Method 2: VPN and Network Interface Resolution

Rating: ⭐⭐⭐⭐ Highly Effective

How it works:

VPN software often modifies network routing tables and DNS configurations, causing Git to attempt connections through incorrect network interfaces. This method systematically identifies and resolves VPN-related connectivity issues by managing network interfaces and routing configurations.

Complete Implementation:

#!/bin/bash
# VPN Network Resolution Script
 
echo "Diagnosing VPN and network interface issues..."
 
# Step 1: Check active network interfaces
echo "Active network interfaces:"
if command -v ip &> /dev/null; then
    ip route show | grep default
elif command -v route &> /dev/null; then
    route -n get default 2>/dev/null || netstat -rn | grep default
else
    echo "Using basic network diagnostics..."
fi
 
# Step 2: Identify VPN interfaces
echo "Checking for VPN interfaces..."
if command -v ip &> /dev/null; then
    ip addr show | grep -E "(tun|tap|ppp|vpn)" || echo "No VPN interfaces detected"
else
    ifconfig 2>/dev/null | grep -E "(tun|tap|ppp|vpn)" || echo "No VPN interfaces detected"
fi
 
# Step 3: Test DNS resolution through different interfaces
echo "Testing DNS resolution..."
nslookup github.com 8.8.8.8 > /dev/null 2>&1
if [ $? -eq 0 ]; then
    echo "✅ DNS resolution working with Google DNS"
else
    echo "❌ DNS resolution failing"
fi
 
# Step 4: VPN-specific Git configuration
echo "Applying VPN-compatible Git settings..."
 
# Disable SSL verification if VPN interferes with certificates
git config --global http.sslVerify false
 
# Set longer timeouts for VPN connections
git config --global http.lowSpeedLimit 1000
git config --global http.lowSpeedTime 300
 
# Use IPv4 only to avoid IPv6 VPN routing issues
git config --global core.gitProxy ""
 
# Step 5: Alternative connection methods
echo "Setting up alternative connection methods..."
 
# Configure SSH as backup
if [ ! -f ~/.ssh/id_rsa ]; then
    echo "Consider setting up SSH keys for more reliable connection"
    echo "Run: ssh-keygen -t rsa -b 4096 -C 'your_email@example.com'"
fi
 
# Test HTTPS connection with custom user agent
git config --global http.userAgent "git/2.0 (compatible; custom-agent)"
 
echo "VPN configuration complete. Test with:"
echo "git clone https://github.com/username/repository.git"

Pros:

  • Addresses VPN-specific routing issues
  • Provides fallback connection methods
  • Maintains VPN security while enabling Git access
  • Includes comprehensive diagnostics

Cons:

  • May disable SSL verification (security consideration)
  • Requires understanding of network interfaces
  • Solutions may be temporary until VPN restart

When to use:

  • When VPN software is active
  • In corporate environments with complex networking
  • When standard proxy reset doesn’t work
  • For remote work scenarios

Method 3: DNS Configuration Fix

Rating: ⭐⭐⭐⭐ Educational & Reliable

How it works:

DNS resolution failures prevent Git from translating “github.com” into an IP address. This method implements multiple DNS resolution strategies, including custom DNS servers, hosts file modifications, and alternative GitHub endpoints.

Complete Implementation:

#!/bin/bash
# DNS Resolution Fix Script
 
echo "Implementing DNS resolution fixes..."
 
# Step 1: Test current DNS resolution
echo "Testing current DNS resolution..."
dig github.com @8.8.8.8 +short > /dev/null 2>&1
if [ $? -eq 0 ]; then
    GITHUB_IP=$(dig github.com @8.8.8.8 +short | head -1)
    echo "✅ GitHub resolves to: $GITHUB_IP"
else
    echo "❌ DNS resolution failing"
    GITHUB_IP="140.82.112.3"  # Fallback GitHub IP
fi
 
# Step 2: Configure Git to use alternative DNS
echo "Configuring alternative DNS resolution..."
 
# Method 2a: Use IP address directly
git config --global url."https://$GITHUB_IP/".insteadOf "https://github.com/"
 
# Method 2b: Configure custom DNS for Git
cat > ~/.gitconfig_dns << EOF
[url "https://140.82.112.3/"]
    insteadOf = https://github.com/
[url "https://140.82.113.3/"]
    insteadOf = https://github.com/
[url "https://140.82.114.3/"]
    insteadOf = https://github.com/
EOF
 
# Step 3: Modify system hosts file (requires admin on Windows)
echo "Checking hosts file configuration..."
HOSTS_FILE="/etc/hosts"
if [[ "$OSTYPE" == "msys" || "$OSTYPE" == "cygwin" ]]; then
    HOSTS_FILE="/c/Windows/System32/drivers/etc/hosts"
fi
 
# Backup hosts file
if [ -f "$HOSTS_FILE" ]; then
    echo "Backing up hosts file..."
    cp "$HOSTS_FILE" "${HOSTS_FILE}.backup.$(date +%Y%m%d)"
fi
 
# Step 4: DNS flush and reset
echo "Flushing DNS cache..."
if command -v systemd-resolve &> /dev/null; then
    sudo systemd-resolve --flush-caches
elif command -v dscacheutil &> /dev/null; then
    sudo dscacheutil -flushcache
elif [[ "$OSTYPE" == "msys" || "$OSTYPE" == "cygwin" ]]; then
    ipconfig //flushdns
fi
 
# Step 5: Alternative GitHub endpoints
echo "Configuring alternative GitHub endpoints..."
git config --global url."https://github.global.ssl.fastly.net/".insteadOf "https://github.com/"
 
# Step 6: SSH alternative setup
echo "Setting up SSH alternative..."
cat > ~/.ssh/config << EOF
Host github.com
    HostName 140.82.112.3
    User git
    Port 22
    IdentityFile ~/.ssh/id_rsa
EOF
 
# Step 7: Test all configurations
echo "Testing DNS configurations..."
for ip in "140.82.112.3" "140.82.113.3" "140.82.114.3"; do
    echo "Testing IP: $ip"
    curl -s --connect-timeout 5 "https://$ip" > /dev/null
    if [ $? -eq 0 ]; then
        echo "✅ $ip is accessible"
        break
    else
        echo "❌ $ip not accessible"
    fi
done
 
echo "DNS configuration complete. Try cloning now."

Pros:

  • Addresses root cause of DNS issues
  • Provides multiple fallback options
  • Educational about network fundamentals
  • Works with restrictive firewalls

Cons:

  • Requires system-level changes
  • May need administrative privileges
  • IP addresses can change over time
  • More complex troubleshooting

When to use:

  • When DNS resolution is clearly failing
  • In environments with restricted DNS servers
  • For learning network troubleshooting
  • When other methods haven’t worked

Method 4: SSL Certificate and Security Settings

Rating: ⭐⭐⭐ Compatibility Focused

How it works:

SSL certificate validation errors can manifest as “could not resolve host” errors when Git fails to establish secure connections. This method addresses certificate chain issues, outdated SSL libraries, and security policy conflicts.

Complete Implementation:

#!/bin/bash
# SSL Certificate and Security Configuration Script
 
echo "Configuring SSL and security settings for Git..."
 
# Step 1: Diagnose SSL issues
echo "Diagnosing SSL connectivity..."
openssl s_client -connect github.com:443 -servername github.com < /dev/null > /dev/null 2>&1
if [ $? -eq 0 ]; then
    echo "✅ SSL connection to GitHub successful"
else
    echo "❌ SSL connection issues detected"
fi
 
# Step 2: Update certificate bundle
echo "Configuring certificate bundle..."
 
# Find system certificate bundle
CERT_BUNDLE=""
for path in "/etc/ssl/certs/ca-certificates.crt" "/etc/pki/tls/certs/ca-bundle.crt" "/usr/local/share/certs/ca-root-nss.crt"; do
    if [ -f "$path" ]; then
        CERT_BUNDLE="$path"
        break
    fi
done
 
if [ -n "$CERT_BUNDLE" ]; then
    git config --global http.sslCAInfo "$CERT_BUNDLE"
    echo "✅ Certificate bundle configured: $CERT_BUNDLE"
else
    echo "⚠️  Certificate bundle not found, using system default"
fi
 
# Step 3: SSL version configuration
echo "Configuring SSL version settings..."
 
# Force TLS 1.2 minimum
git config --global http.sslVersion "tlsv1.2"
 
# Configure cipher suites for compatibility
git config --global http.sslCipherList "ECDHE+AESGCM:ECDHE+AES256:ECDHE+AES128:!aNULL:!MD5:!DSS"
 
# Step 4: Corporate certificate handling
echo "Configuring corporate certificate handling..."
 
# Check for corporate certificates
if [ -d "/usr/local/share/ca-certificates" ]; then
    echo "Corporate certificate directory found"
    # Update certificate store
    if command -v update-ca-certificates &> /dev/null; then
        sudo update-ca-certificates
    fi
fi
 
# Step 5: Fallback SSL configurations
echo "Setting up SSL fallback configurations..."
 
# Create SSL configuration script
cat > ~/.git_ssl_config << 'EOF'
#!/bin/bash
# Git SSL Configuration Fallbacks
 
# Configuration 1: Standard SSL
configure_standard_ssl() {
    git config --global http.sslVerify true
    git config --global http.sslVersion "tlsv1.2"
}
 
# Configuration 2: Relaxed SSL (for corporate environments)
configure_relaxed_ssl() {
    git config --global http.sslVerify false
    git config --global http.sslVersion "tlsv1.0"
}
 
# Configuration 3: Custom certificate
configure_custom_cert() {
    if [ -f "$1" ]; then
        git config --global http.sslCAInfo "$1"
        git config --global http.sslVerify true
    fi
}
 
# Test function
test_git_connection() {
    echo "Testing Git connection..."
    git ls-remote https://github.com/octocat/Hello-World.git > /dev/null 2>&1
    return $?
}
 
# Auto-configure based on environment
auto_configure() {
    echo "Auto-configuring SSL settings..."
    
    # Try standard configuration first
    configure_standard_ssl
    if test_git_connection; then
        echo "✅ Standard SSL configuration working"
        return 0
    fi
    
    # Try relaxed configuration
    echo "Trying relaxed SSL configuration..."
    configure_relaxed_ssl
    if test_git_connection; then
        echo "⚠️  Relaxed SSL configuration working (less secure)"
        return 0
    fi
    
    echo "❌ SSL configuration failed"
    return 1
}
 
# Run auto-configuration if script is executed
if [ "${BASH_SOURCE[0]}" == "${0}" ]; then
    auto_configure
fi
EOF
 
chmod +x ~/.git_ssl_config
 
# Step 6: Run SSL auto-configuration
echo "Running SSL auto-configuration..."
bash ~/.git_ssl_config
 
# Step 7: Alternative protocols
echo "Configuring alternative protocols..."
 
# Enable HTTP/2 if available
git config --global http.version "HTTP/2"
 
# Configure connection pooling
git config --global http.maxRequests 10
 
echo "SSL configuration complete."
echo "If issues persist, run: bash ~/.git_ssl_config"

Pros:

  • Addresses certificate validation issues
  • Provides multiple security configurations
  • Compatible with corporate environments
  • Includes automatic fallback options

Cons:

  • May reduce security in some configurations
  • Requires understanding of SSL/TLS
  • Corporate policies may override settings
  • Complex troubleshooting process

When to use:

  • In corporate environments with custom certificates
  • When SSL errors are reported in verbose output
  • For older systems with outdated SSL libraries
  • When security policies interfere with Git

Method 5: Complete Network Reset

Rating: ⭐⭐⭐⭐⭐ Nuclear Option (Always Works)

How it works:

This comprehensive approach resets all network-related configurations and rebuilds Git’s network stack from scratch. It’s the most thorough solution that addresses multiple potential issues simultaneously.

Complete Implementation:

#!/bin/bash
# Complete Network Reset Script for Git
 
echo "Performing complete network reset for Git..."
echo "⚠️  This will reset all Git network configurations"
 
# Step 1: Backup current configuration
echo "Backing up current Git configuration..."
cp ~/.gitconfig ~/.gitconfig.backup.$(date +%Y%m%d_%H%M%S) 2>/dev/null || echo "No existing .gitconfig"
 
# Step 2: Reset all Git network configurations
echo "Resetting Git network configurations..."
git config --global --unset-all http.proxy 2>/dev/null
git config --global --unset-all https.proxy 2>/dev/null
git config --global --unset-all http.sslVerify 2>/dev/null
git config --global --unset-all http.sslCAInfo 2>/dev/null
git config --global --unset-all http.sslVersion 2>/dev/null
git config --global --unset-all http.userAgent 2>/dev/null
git config --global --unset-all credential.helper 2>/dev/null
git config --global --unset-all core.gitProxy 2>/dev/null
git config --global --unset-all url.https://github.com/.insteadOf 2>/dev/null
 
# Step 3: Clear Git credential cache
echo "Clearing Git credential cache..."
if command -v git-credential-manager &> /dev/null; then
    git-credential-manager delete --url https://github.com 2>/dev/null
fi
 
if command -v git-credential-store &> /dev/null; then
    rm -f ~/.git-credentials 2>/dev/null
fi
 
# Step 4: Network interface reset
echo "Resetting network interfaces..."
 
# Flush DNS cache
if command -v systemd-resolve &> /dev/null; then
    sudo systemd-resolve --flush-caches
elif command -v dscacheutil &> /dev/null; then
    sudo dscacheutil -flushcache
    sudo killall -HUP mDNSResponder
elif [[ "$OSTYPE" == "msys" || "$OSTYPE" == "cygwin" ]]; then
    ipconfig //flushdns
    ipconfig //release
    ipconfig //renew
fi
 
# Step 5: Reset network stack (platform-specific)
echo "Resetting network stack..."
 
if [[ "$OSTYPE" == "linux-gnu"* ]]; then
    # Linux network reset
    sudo systemctl restart NetworkManager 2>/dev/null || sudo service network-manager restart 2>/dev/null
elif [[ "$OSTYPE" == "darwin"* ]]; then
    # macOS network reset
    sudo dscacheutil -flushcache
    sudo killall -HUP mDNSResponder
elif [[ "$OSTYPE" == "msys" || "$OSTYPE" == "cygwin" ]]; then
    # Windows network reset
    netsh winsock reset
    netsh int ip reset
fi
 
# Step 6: Rebuild Git configuration with optimal settings
echo "Rebuilding Git configuration with optimal settings..."
 
# Basic HTTP settings
git config --global http.lowSpeedLimit 1000
git config --global http.lowSpeedTime 300
git config --global http.maxRequests 10
 
# SSL settings
git config --global http.sslVersion "tlsv1.2"
git config --global http.sslVerify true
 
# Connection settings
git config --global http.version "HTTP/2"
 
# Step 7: Test multiple connection methods
echo "Testing multiple connection methods..."
 
# Test 1: Basic HTTPS
echo "Testing HTTPS connection..."
git ls-remote https://github.com/octocat/Hello-World.git > /dev/null 2>&1
if [ $? -eq 0 ]; then
    echo "✅ HTTPS connection working"
    HTTPS_WORKING=true
else
    echo "❌ HTTPS connection failed"
    HTTPS_WORKING=false
fi
 
# Test 2: SSH (if keys exist)
if [ -f ~/.ssh/id_rsa ] || [ -f ~/.ssh/id_ed25519 ]; then
    echo "Testing SSH connection..."
    ssh -T git@github.com > /dev/null 2>&1
    if [ $? -eq 1 ]; then  # SSH returns 1 for successful auth test
        echo "✅ SSH connection working"
        SSH_WORKING=true
    else
        echo "❌ SSH connection failed"
        SSH_WORKING=false
    fi
else
    echo "⚠️  No SSH keys found"
    SSH_WORKING=false
fi
 
# Step 8: Provide connection recommendations
echo ""
echo "=== CONNECTION TEST RESULTS ==="
if [ "$HTTPS_WORKING" = true ]; then
    echo "✅ HTTPS: Use git clone https://github.com/username/repo.git"
elif [ "$SSH_WORKING" = true ]; then
    echo "✅ SSH: Use git clone git@github.com:username/repo.git"
else
    echo "❌ Both HTTPS and SSH failed. Additional troubleshooting needed."
    echo ""
    echo "=== ADDITIONAL TROUBLESHOOTING ==="
    echo "1. Check firewall settings"
    echo "2. Verify internet connectivity: ping 8.8.8.8"
    echo "3. Test GitHub accessibility: curl -I https://github.com"
    echo "4. Check corporate proxy requirements"
    echo "5. Contact network administrator"
fi
 
# Step 9: Create recovery script
echo "Creating recovery script..."
cat > ~/.git_recovery << 'EOF'
#!/bin/bash
# Git Recovery Script - Restore from backup
 
BACKUP_FILE=$(ls ~/.gitconfig.backup.* 2>/dev/null | tail -1)
 
if [ -n "$BACKUP_FILE" ]; then
    echo "Restoring Git configuration from: $BACKUP_FILE"
    cp "$BACKUP_FILE" ~/.gitconfig
    echo "✅ Configuration restored"
else
    echo "❌ No backup file found"
fi
EOF
 
chmod +x ~/.git_recovery
 
echo ""
echo "=== NETWORK RESET COMPLETE ==="
echo "If you need to restore your previous configuration:"
echo "bash ~/.git_recovery"
echo ""
echo "Network reset complete. Try cloning your repository now."

Pros:

  • Addresses all possible network configuration issues
  • Provides comprehensive diagnostics
  • Creates backup and recovery options
  • Works across all platforms

Cons:

  • Resets all Git network configurations

  • Overkill for simple issues

Common Problems & Solutions

Problem 1: “SSL certificate problem: unable to get local issuer certificate”

Solution: This indicates SSL certificate validation issues, often in corporate environments.

# Temporary fix (less secure)
git config --global http.sslVerify false
 
# Permanent fix - configure proper certificate bundle
git config --global http.sslCAInfo /path/to/certificate/bundle.crt
 
# For corporate environments
git config --global http.sslCAInfo /etc/ssl/certs/ca-certificates.crt

Problem 2: “Failed to connect to github.com port 443: Connection timed out”

Solution: This indicates firewall or network routing issues.

# Test different ports
git config --global url."https://github.com:443/".insteadOf "https://github.com/"
 
# Use SSH on port 22 instead
git config --global url."ssh://git@github.com:22/".insteadOf "https://github.com/"
 
# Configure SSH to use port 443 (if 22 is blocked)
echo "Host github.com
    HostName ssh.github.com
    Port 443
    User git" >> ~/.ssh/config

Problem 3: “fatal: unable to access ‘https://github.com/ ’: Received HTTP code 407 from proxy”

Solution: This indicates proxy authentication is required.

# Configure proxy with authentication
git config --global http.proxy http://username:password@proxy.company.com:8080

Frequently Asked Questions

Q: Which method should I choose?

A: Start with Method 1 (Proxy Reset) as it solves 80% of cases in under a minute. If you’re in a corporate environment, try Method 2 (VPN Resolution) next. For persistent issues, Method 5 (Complete Reset) provides the most comprehensive solution.

Q: What about browser compatibility?

A: Git network issues are independent of browser compatibility. However, if you can access GitHub in your browser but Git fails, it indicates Git-specific configuration problems rather than general network issues.

Q: How do I handle edge cases?

A: Edge cases often require combining methods. For example, corporate environments might need both proxy configuration (Method 1) and SSL certificate handling (Method 4). The Complete Reset (Method 5) addresses multiple edge cases simultaneously.

Q: Are there performance differences?

A: Yes, significant differences exist:

  • Method 1: Fastest resolution (30 seconds)
  • Method 3: May improve long-term performance with optimized DNS
  • Method 5: Slower initially but provides most stable long-term performance
  • SSH alternatives generally perform better than HTTPS in restricted networks

Q: Can I combine multiple approaches?

A: Absolutely. Many enterprise environments require combining proxy configuration, SSL certificate management, and DNS optimization. The key is applying them in the correct order and testing each step.
